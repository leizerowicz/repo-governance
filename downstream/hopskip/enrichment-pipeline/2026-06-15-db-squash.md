## Background

enrichment-pipeline has 62 migration files spanning 2025-10-25 through 2026-06-13. The governance standard triggers a squash at 50 non-baseline increments. This prompt runs the squash and ensures the existing harness is gated on PRs — not just on prod deploys.

## What to do

### 1. Run the squash

The `dump-schema --minimal` command already exists. From the `dbmigrations/` directory:

```powershell
dotnet run -- dump-schema --minimal \
  -h <prod-or-staging-host> -u <user> -w <password> -d enrichment_pipeline_db \
  -f scripts/migrations/20260615_0001_squash_baseline.sql
```

Curate the output file:
- Add `IF NOT EXISTS` guards to every `CREATE TABLE`, `CREATE INDEX`, `CREATE FUNCTION`, `CREATE VIEW` statement — the dump may produce bare `CREATE` statements
- Add `CREATE OR REPLACE` for functions and views where supported
- Add the required header block at the top:

```sql
-- Squash baseline: 20260615
-- Replaces: 20251025_0001_extensions_prereqs.sql through 20260613_0001_add_publishable_column.sql (62 files)
-- Trigger: b — 62 files exceeded the 50-file governance threshold
-- Applied to all environments as of: 2026-06-15
-- New migrations resume after this date prefix
```

Verify the baseline covers everything: compare `CREATE TABLE` count in the baseline against the table count from `information_schema.tables` on the live DB.

### 3a. Mark baseline as pre-applied in all existing environments (post-merge)

After the squash PR merges, run this against every live environment (prod, staging, qa) before re-triggering the migration workflow. DbUp has never seen the baseline filename and will attempt to apply it — even with `IF NOT EXISTS` on tables, constraints like primary keys added separately will fail with "already exists". Pre-registering in `schemaversions` is the safe fix:

```sql
INSERT INTO public.schemaversions (scriptname, applied)
SELECT 'dbmigrations.scripts.migrations.20260615_0001_squash_baseline.sql', NOW()
WHERE NOT EXISTS (
  SELECT 1 FROM public.schemaversions
  WHERE scriptname = 'dbmigrations.scripts.migrations.20260615_0001_squash_baseline.sql'
);
```

Verify per environment before re-running the workflow:

```sql
SELECT scriptname FROM schemaversions WHERE scriptname LIKE '%squash_baseline%';
-- must return one row
```

Then re-trigger the migration workflow — DbUp will see the baseline in the journal and skip it cleanly.

### 4. Verify the harness passes

```powershell
dotnet run -- test-harness --spawn-container
```

This must pass cleanly against the baseline before the squash PR merges. If verification scripts reference object names that changed, update them now.

### 5. Add the PR gate workflow

Create `.github/workflows/db-migration-harness.yml`. Copy from `~/repos/greg/repo-governance/templates/workflows/db-migration-harness-postgres.yml` and replace `<DBMIGRATIONS_DIR>` with `dbmigrations`.

The existing `run-db-migrations.yml` triggers on push to main and runs the harness before prod deploy — that stays. This new workflow is the PR-time gate that catches problems before merge.

### 6. Register as a required status check

In GitHub → Settings → Branches → main → Branch protection rules:

- Add `Migration harness (ephemeral Postgres)` (the job name from the new workflow) to **Required status checks**
- Ensure "Require status checks to pass before merging" is enabled
